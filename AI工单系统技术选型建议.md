# AI工单系统技术选型与开发流程（基于企业级 RAG）

## 1. 背景与目标

当前运维与客户支持场景中，客户遇到问题后通常先提交工单，但处理链路高度依赖人工分发和经验判断，响应速度与处理一致性难以保障。  
很多问题本质上是可标准化复用的问题，但由于缺少智能分流与知识沉淀机制，常出现“重复提问、重复解答、跨团队反复流转”的情况。

本方案目标是建设一个可上线的 AI 工单系统，形成清晰闭环：

- 客户提问后，AI优先基于知识库自动回答
- AI无法有效回答时，自动转一线人工客服继续处理
- 一线无法闭环的问题，升级转二线人工技术客服
- 对高质量、高复用价值的问题与答案执行入库沉淀
- 持续提升自动解决率与整体服务效率

---

## 2. 总体架构（双管道）

采用离线索引管道 + 在线查询管道的 RAG 架构，并叠加工单流程引擎。

### 2.1 离线索引管道

文档上传 -> 文档解析 -> 分块 -> Embedding -> 存储 PGVector

能力要点：

- 支持 PDF/Word/Markdown/HTML/表格文档
- 文档版本管理（新版本覆盖、历史可追溯）
- 差量更新（仅重建变更文档向量）
- 异步任务化（解耦上传与索引）

### 2.2 在线查询管道

查询改写 -> 混合检索 -> RRF 融合 -> Reranker 精排 -> 上下文裁剪 -> 引用溯源 -> 流式生成

能力要点：

- 检索链路每一环支持独立监控和降级
- 回答必须携带引用来源，支持点击溯源
- 对低置信度问题触发“补问 + 转人工”策略

## 3. 技术栈选型（与你给出的方案对齐）

- 向量模型：`text-embedding-v3`
- 重排模型：`gte-rerank`

说明：  
该组合在中文企业知识问答场景下可兼顾语义理解、检索召回与输出可读性，且具备较好的工程集成成熟度。

## 4. 四个关键技术决策（建议保留）

## 4.1 自定义检索编排，不依赖默认 QuestionAnswerAdvisor

不采用默认 Advisor 一把梭，而是将在线链路拆为可控步骤。原因：

- 默认能力难覆盖企业场景的混合检索、重排、引用溯源
- 无法在每一步设置细粒度降级策略
- 链路黑盒，不利于问题定位、性能优化和效果评估

选型结论：  
以“多写代码换可控性”是正确取舍，尤其适合工单系统这种强 SLA 场景。

## 4.2 混合检索 + RRF 融合

仅向量检索对“专有词/错误码/条款号”召回不足，仅全文检索对语义改写问题又会漏召回。  
采用双路检索并融合是企业落地的常见最优解。

建议参数：

- 向量检索：HNSW，Top 20
- 全文检索：PostgreSQL GIN + tsvector，Top 20
- 融合算法：RRF，k=60

收益：

- 显著提升精确词问题召回率（如 `SKU-8821`、`第3.2条`）
- 不依赖分数归一化，工程复杂度低

## 4.3 权限在检索层硬过滤，不依赖 Prompt

多租户/多部门场景下，权限必须在数据访问层强制生效。  
正确做法是在查询阶段加入白名单过滤，例如：

- `WHERE kb_id IN (:allowedKbIds)`

`allowedKbIds` 由 `PermissionService` 返回，与模型解耦。  
这样即使出现 Prompt 注入，模型也拿不到未授权数据。

## 4.4 Reranker 超时降级

外部重排服务存在波动时，不能拖垮主链路。  
建议机制：

- `gte-rerank` 超时阈值 800ms
- 超时后降级为 RRF 结果直接截断 Top 5
- 保证系统可用性优先于局部精度

这套策略可显著提升线上稳定性，对工单系统非常关键。

## 5. 面向工单系统的功能架构补全

在上述 RAG 能力基础上，需要补齐工单域能力，形成可运营闭环。

## 5.1 工单状态机

建议标准状态：

- `新建`
- `AI处理中`
- `待补充信息`
- `转一线`
- `转二线`
- `已解决`
- `已关闭`

每次状态流转必须记录：触发规则、责任人、时间戳、处理备注。

## 5.2 分流与升级策略

- FAQ高置信度：AI直接回复并建议用户确认关闭
- 低置信度：先补问关键信息（版本、环境、报错截图、影响范围）
- 仍无法定位：自动转一线
- 涉及代码缺陷、架构问题或跨系统故障：转二线
- P1/P2故障可配置“跳过AI，直接人工”

## 5.3 知识回流机制

- 已解决工单自动生成“候选知识条目”
- 审核通过后入正式知识库
- 标记来源工单、版本、适用范围
- 对高频未解决问题建立周复盘机制

## 6. 开发流程（建议 10 周）

## 阶段0：需求与数据盘点（第1周）

- 明确工单类型与 SLA 分级规则
- 梳理知识源目录和权限模型（部门/角色/租户）
- 确定验收指标（自动解决率、响应延迟、转人工准确率）

交付物：

- PRD v1
- 工单状态机与流转规则文档
- 知识源清单与权限矩阵

## 阶段1：离线索引管道（第2-3周）

- 打通上传 -> 解析 -> 分块 -> Embedding -> PGVector
- 建立文档版本管理与差量更新
- 建立 MinIO 文档存储与元数据表

交付物：

- 可运行索引任务服务
- 文档处理结果看板（成功率/耗时）

## 阶段2：在线检索链路（第4-5周）

- 实现查询改写、双路检索、RRF融合、Reranker、上下文裁剪
- 完成引用溯源与流式输出
- 增加 Reranker 超时降级策略

交付物：

- 在线问答 API
- 检索效果基线报告（TopK命中率）

## 阶段3：工单流程集成（第6-7周）

- 接入工单创建、评论、状态流转、转派能力
- 实现 AI -> 一线 -> 二线 升级链路
- 接入转单原因记录与 SLA 计时

交付物：

- 工单全链路可用
- 升级与派单策略配置中心

## 阶段4：安全与多租户（第8周）

- 检索层权限硬过滤全链路落地
- 审计日志（检索条件、引用来源、答案版本）
- 敏感信息脱敏与访问追踪

交付物：

- 安全测试报告
- 权限回归测试用例集

## 阶段5：评估、压测、灰度（第9-10周）

- 评估流水线：召回率、精排收益、答案准确率、幻觉率
- 性能压测：QPS、P95延迟、降级触发比例
- 成本看板：Token、Embedding、Reranker调用成本
- 小范围灰度上线并复盘

交付物：

- 上线评审材料
- 灰度复盘报告
- 生产发布清单与回滚预案

---

## 7. 关键指标与验收标准

建议在立项阶段冻结以下指标：

- 自动解决率（AI直接解决）>= 35%
- 首次响应时间（P95）<= 10s
- 检索命中率（Top5）>= 85%
- 转人工准确率 >= 90%
- 引用覆盖率（回答附来源）>= 95%
- 权限误召回（越权）= 0

---

## 8. 风险与应对

- 检索质量不足：增加查询改写策略与知识清洗规范
- 外部模型抖动：设置超时、熔断、降级、重试策略
- 权限配置复杂：权限中心统一收口，检索层强约束
- 知识更新滞后：建立“工单结案 -> 候选知识”自动回流
- 成本上升：增加缓存、批量Embedding、调用配额与告警

---

## 9. 最终建议（可直接用于决策）

1. 你的方案可直接用于 AI 工单系统，方向正确且具备企业落地特征。  
2. 保留四个核心技术点，不建议回退到默认黑盒链路。  
3. 优先完成“检索可控 + 权限硬隔离 + 降级可用”，再优化模型表现。  
4. 将工单状态机和 SLA 引擎作为与 RAG 同等优先级模块推进。  
5. 先从一个业务部门灰度上线，跑通闭环后再全公司推广。
